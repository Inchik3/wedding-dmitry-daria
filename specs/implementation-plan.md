# Implementation Plan

Реализация начинается только после точной команды: «Спеки утверждены, приступай к реализации».

## 1. Инициализация проекта

- Цель: создать Vite + TypeScript проект без React/Vue/Angular.
- Файлы: `package.json`, `index.html`, `src/main.ts`, configs.
- Зависимости: Vite, TypeScript.
- Проверка: dev server запускается, пустая страница открывается.
- Риски: случайно создать лишний фреймворк.
- Критерий: минимальный проект создан.

## 2. Базовая архитектура

- Цель: подготовить каталоги `src/config`, `src/modules`, `src/shared`.
- Файлы: структура модулей и базовые types.
- Зависимости: этап 1.
- Проверка: typecheck проходит.
- Риски: переусложнение.
- Критерий: модули разделены по ответственности.

## 3. Дизайн-токены и глобальные стили

- Цель: внедрить палитру, типографику, layout tokens.
- Файлы: `src/styles/base.css`, `components.css`, `responsive.css`.
- Зависимости: этап 2.
- Проверка: базовые секции визуально стабильны на 320px и desktop.
- Риски: слишком бледный контраст.
- Критерий: стили покрывают базовые элементы и states.

## 4. Конверт

- Цель: реализовать стартовый overlay и анимацию открытия.
- Файлы: HTML-разметка, `envelope.ts`, CSS.
- Зависимости: этап 3.
- Проверка: открывается кликом и клавиатурой, фокус переносится.
- Риски: overlay блокирует доступность.
- Критерий: основной сайт доступен после действия.

## 5. Hero и фотозаглушки

- Цель: hero с `hero.webp` или placeholder.
- Файлы: HTML, CSS, image helpers.
- Зависимости: этап 3.
- Проверка: фото/placeholder сохраняют композицию.
- Риски: текст плохо читается на фото.
- Критерий: hero работает без реального изображения.

## 6. Основные секции

- Цель: приветствие, программа, places, dress-code, final.
- Файлы: HTML, `content.ts`, CSS.
- Зависимости: этапы 3-5.
- Проверка: тексты совпадают с `content.md`.
- Риски: добавить лишние события.
- Критерий: все смысловые блоки есть и не выходят за scope.

## 7. Таймер

- Цель: обратный отсчет до `2026-08-28T13:00:00+03:00`.
- Файлы: `countdown.ts`, `time.ts`, unit tests.
- Зависимости: этап 2.
- Проверка: Vitest покрывает до/после даты и timezone.
- Риски: зависимость от локального timezone.
- Критерий: расчет основан на абсолютном timestamp.

## 8. Карты

- Цель: две Яндекс Карты без API-ключа, fallback и маршруты.
- Файлы: `maps.ts`, HTML, CSS.
- Зависимости: этап 6.
- Проверка: адреса и кнопки работают без iframe.
- Риски: iframe URL нестабилен.
- Критерий: fallback остается полезным.

## 9. Дресс-код

- Цель: блок палитры, swatches, будущий анализ изображения.
- Файлы: CSS tokens, HTML, README-инструкции.
- Зависимости: этап 3.
- Проверка: цвета не нарушают контраст.
- Риски: файл палитры отсутствует.
- Критерий: блок корректен с placeholder и после добавления файла.

## 10. RSVP frontend

- Цель: форма, состояния и client validation.
- Файлы: `rsvp.ts`, `validation.ts`, CSS, tests.
- Зависимости: этап 6.
- Проверка: обязательные поля, напитки, отказ.
- Риски: потеря введенных данных при ошибке.
- Критерий: форма валидируется локально.

## 11. Google Apps Script

- Цель: endpoint для записи RSVP.
- Файлы: `google-apps-script/Code.gs`.
- Зависимости: утвержденный API contract.
- Проверка: ручной тест deployed web app.
- Риски: CORS и ограничения Apps Script.
- Критерий: JSON ответ и запись в Sheets.

## 12. Интеграция формы

- Цель: отправка в Apps Script с retry/error/success.
- Файлы: API client, `rsvp.ts`, `.env.example`.
- Зависимости: этапы 10-11.
- Проверка: mocked endpoint и отсутствующий URL.
- Риски: network errors.
- Критерий: UX устойчив к ошибкам.

## 13. Accessibility pass

- Цель: клавиатура, ARIA, focus, reduced-motion.
- Файлы: HTML, CSS, modules.
- Зависимости: все UI этапы.
- Проверка: ручная клавиатурная навигация и Playwright.
- Риски: overlay/focus trap.
- Критерий: путь клавиатурой полный.

## 14. Responsive pass

- Цель: проверить 320, 390, 768, 1024, 1440px.
- Файлы: CSS.
- Зависимости: все UI этапы.
- Проверка: screenshots/manual.
- Риски: переполнение текста.
- Критерий: нет горизонтального scroll и наложений.

## 15. Performance pass

- Цель: оптимизировать загрузку.
- Файлы: HTML, CSS, TS.
- Зависимости: ассеты/карты.
- Проверка: build size, lazy loading, Lighthouse ориентир.
- Риски: тяжелые карты.
- Критерий: основной контент доступен при медленной сети.

## 16. Unit tests

- Цель: Vitest для логики.
- Файлы: `tests/unit`.
- Зависимости: модули countdown/validation/api.
- Проверка: `npm run test`.
- Риски: тестировать DOM вместо логики.
- Критерий: критичная логика покрыта.

## 17. E2E tests

- Цель: Playwright сценарии.
- Файлы: `tests/e2e`.
- Зависимости: UI готов.
- Проверка: `npm run test:e2e`.
- Риски: flaky animations.
- Критерий: основные пользовательские сценарии проходят.

## 18. CI

- Цель: GitHub Actions для typecheck/test/build/deploy.
- Файлы: `.github/workflows/pages.yml`.
- Зависимости: tests/build.
- Проверка: локально команды проходят, workflow синтаксически корректен.
- Риски: Playwright в CI.
- Критерий: workflow готов к push.

## 19. GitHub Pages

- Цель: base path и публикация `dist`.
- Файлы: Vite config, workflow, README.
- Зависимости: CI.
- Проверка: build с `VITE_BASE_PATH=/repository-name/`.
- Риски: сломанные пути ассетов.
- Критерий: preview/build корректны с base.

## 20. README

- Цель: инструкции для заказчика.
- Файлы: `README.md`.
- Зависимости: все решения финализированы.
- Проверка: инструкции покрывают фото, палитру, музыку, Apps Script, Pages.
- Риски: неполные шаги публикации.
- Критерий: проект можно обслуживать без повторного объяснения.

## 21. Финальная проверка

- Цель: выполнить TypeScript typecheck, Vitest, Playwright, `npm run build`.
- Файлы: отчет в финальном ответе.
- Зависимости: вся реализация.
- Проверка: фактические команды.
- Риски: недоступные внешние карты/Apps Script.
- Критерий: честный финальный отчет с тем, что проверено и что нет.

## Обновление плана реализации: рукописная версия

Следующие этапы выполняются только после отдельного утверждения обновленных спецификаций.

1. Обновить контент и конфигурацию:

   - сократить палитру до 6 оттенков;
   - добавить `arrivalTime` варианты;
   - добавить пути `groom.webp` и `bride.webp`.

2. Обновить типографику:
   - подобрать каллиграфический шрифт с кириллицей;
   - подключить его легально;

3. Пересобрать первый экран:
   - убрать фото из основного hero-варианта;
   - создать молочно-шелковый CSS-фон;
   - добавить reduced-motion fallback.

4. Переработать таймер:
   - легкая горизонтальная строка desktop;
   - две строки на mobile;

5. Переработать программу:

7. Переработать дресс-код:
   - новый текст;
   - 6 оттенков;
   - компактные сердечки/тканевые плашки/овалы вместо крупных кругов.

8. Обновить RSVP:
   - добавить обязательный radio-вопрос «Ко скольки мы будем вас ждать?»;
   - показывать только при присутствии;

   - не добавлять телефон.

9. Обновить API и Apps Script:
   - добавить `arrivalTime`;
   - обновить порядок столбцов;
   - добавить «Просроченный ответ»;
   - обновить server-side validation.

10. Обновить тесты:
    - unit: validation для arrivalTime;
    - e2e: показ/скрытие arrivalTime, обязательность для присутствующих, отсутствие для отказа;
    - e2e/visual smoke: hero без фото, палитра 6 элементов, два события программы.

11. Обновить документацию и README.

12. Выполнить проверки и деплой только после подтверждения:
    - `pnpm typecheck`;
    - `pnpm test`;
    - `pnpm build`;
    - `pnpm test:e2e`;
